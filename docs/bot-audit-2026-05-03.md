## 1. CAPTURE LOGIC PER BOT

**All three bots share the same capture evaluation function.** There is no per-bot capture scanning code. The function `evaluateAllActions()` in `src/engine/ai/evaluator.ts:327-403` runs for all bots identically. It:

1. For each card in the bot's hand, finds all legal single captures via `findAllCaptures(handCard, board)`
2. For each single capture, scores it across 6 dimensions via `scoreCapture()` (evaluator.ts:108-178)
3. If `allowMultiSlot` is true for the personality, also evaluates multi-slot captures via `findBestMultiSlotCapture()`
4. Scores each placement option via `scorePlacement()` (evaluator.ts:181-231)
5. Applies personality weights to each dimension, sums to a total score
6. Sorts all actions by total score descending

**What differentiates bots is the WEIGHT applied to each dimension, not the scanning logic.**

### Calvin (beginner)
- File: `src/engine/ai/personalities/calvin.ts`
- Weights: `rawPoints: 1.0`, all others 0.0 except `placementDanger: 0.1`
- `allowMultiSlot: false` — Calvin only evaluates single-slot captures
- `useChainEval: false` — no lookahead
- `mistakeRate: 0.25` — 25% of the time, picks randomly from top 50% of actions instead of best
- Net effect: Calvin is a greedy point-maximizer who sometimes makes mistakes and never finds multi-area combos

### Nina (intermediate)
- File: `src/engine/ai/personalities/nina.ts`
- Weights: `rawPoints: 1.0, chainPotential: 0.5, placementDanger: 0.7, opponentDenial: 0.3, jackpotValue: 0.4, boardControl: 0.3`
- `allowMultiSlot: true` — Nina evaluates multi-slot captures
- `useChainEval: false` — no 2-turn lookahead
- `preferSumsOnTie: true` — when two captures have equal total score, Nina prefers the one using sum captures over pairs (botDecision.ts:77-87)
- `mistakeRate: 0.08` — 8% mistake rate
- Net effect: balanced play, considers denial and board control but doesn't plan ahead

### Rex (advanced)
- File: `src/engine/ai/personalities/rex.ts`
- Weights: `rawPoints: 0.8, chainPotential: 1.0, placementDanger: 1.0, opponentDenial: 0.9, jackpotValue: 1.0, boardControl: 0.8`
- `allowMultiSlot: true` — Rex evaluates multi-slot captures
- `useChainEval: true` — Rex evaluates 2-turn chain captures via `evaluateChainCapture()` (evaluator.ts:260-304). If a 2-turn chain scores 20%+ better than the best single capture (`plan.totalPoints > bestCaptureTotal * 1.2`), Rex plays the chain's first move instead (botDecision.ts:161-182)
- `mistakeRate: 0.02` — 2% mistake rate
- Net effect: strategic, plans chains, heavily weights denial and jackpot

### Multi-area combo evaluation
Yes — Nina and Rex (`allowMultiSlot: true`) evaluate multi-slot captures. Calvin does not.

### Lookahead / chain detection
Only Rex (`useChainEval: true`). The chain evaluator tries all hand cards as a first capture, simulates the resulting board, then tries remaining hand cards for a second capture. Returns the best 2-turn total. Implemented in `evaluateChainCapture()` at evaluator.ts:260-304.

## 2. PLACEMENT LOGIC PER BOT

**All three bots share the same placement scoring function** `scorePlacement()` at evaluator.ts:181-231. The function calculates a `placementDanger` score for each card in hand by:

1. For each rank in the deck, checking how many remain unseen (via CardTracker)
2. If the rank matches the card being placed → danger += SCORE_VALUES[rank] (opponent could pair it)
3. If an opponent could use a higher-value card to sum-capture with this placed card and existing board cards → danger += SCORE_VALUES[rank]
4. In endgame, if bot is not lastCapturer, adds -10 jackpot penalty

The danger score is NEGATED (multiplied by -1) before weighting, so higher danger = lower total score = less likely to be chosen.

### Calvin's "tell" — placing highest card
**YES, confirmed implemented.** In `botDecision.ts:195-206`:
```
if (profile.preferHighestValueOnPlace && chosen.action === 'place') {
    const highest = calvinPlacementPick(state.hands[playerIndex]);
    if (highest.id !== chosen.handCard.id) {
      const swap = actions.find(a => a.action === 'place' && a.handCard.id === highest.id);
      if (swap) chosen = swap;
    }
}
```
Calvin's profile has `preferHighestValueOnPlace: true`. The `calvinPlacementPick()` function (evaluator.ts:451-463) finds the card with the highest `calculateCardPoints()` value. This OVERRIDES the normal danger-based placement selection.

### Nina avoiding dangerous placements
Yes, but only via weighting. Nina's `placementDanger` weight is 0.7, meaning she weighs placement risk at 70% of full strength. She does NOT have a hard gate or filter — she just scores dangerous placements lower. If a dangerous placement still has the highest total score (e.g., no captures available and all placements are dangerous), she'll place the least-dangerous option.

### Rex baiting
**Not implemented.** There is no "setup value" calculation — Rex does not check whether placing a card creates a capture opportunity for himself on the next turn. This was identified in the AI audit (docs/AI_AUDIT_REPORT.md) as a gap vs. the LIVE version's Rex, which had `_calculateSetupValue()`. The v2 Rex only avoids danger on placement, never seeks opportunity.

### Are placement strategies different per bot?
They differ by WEIGHT, not by logic:
- Calvin: `placementDanger: 0.1` (barely considers danger) + `preferHighestValueOnPlace: true` (overrides to always place highest)
- Nina: `placementDanger: 0.7` (moderate danger avoidance)
- Rex: `placementDanger: 1.0` (full danger avoidance)

## 3. RISK THRESHOLD / VALUE GATE

### Minimum capture value threshold
**Not implemented.** No bot has a minimum point threshold for taking a capture. If any capture scores higher than the best placement (after weighting), the bot takes it regardless of how small. A 5-point pair capture will be taken if it beats all placement options.

### Percentage-based thresholds (architectural rule)
**YES, confirmed followed** for the dynamic weight modifiers. In `modifyWeightsForGameState()` at evaluator.ts:413-449:

```typescript
const target = state.settings.targetScore;
if (!target || target <= 0) return baseWeights;
```

Rex denial mode: `if (maxOpponent >= target * 0.50)` — triggers at 50% of target  
Rex conservative: `if (myScore - maxOpponent >= target * 0.20)` — triggers at 20% lead  
Nina aggressive: `if (maxOpponent - myScore >= target * 0.15)` — triggers at 15% deficit

No hardcoded absolute values. All three thresholds scale with targetScore.

### Actual percentage thresholds

| Bot | Mode | Condition | Modified weights |
|-----|------|-----------|-----------------|
| Rex | Denial | Any opponent >= 50% of target | opponentDenial: 1.8, boardControl: 1.2 |
| Rex | Conservative | Rex leads by >= 20% of target | rawPoints: 0.5, placementDanger: 1.5, opponentDenial: 0.45 |
| Nina | Aggressive | Behind by >= 15% of target | rawPoints: 1.3, opponentDenial: 0.5, placementDanger: 0.5 |
| Calvin | (none) | N/A — always returns base weights | No modification |

Code location: evaluator.ts:413-449

## 4. WHAT SCALES WITH DIFFICULTY

### Does bot behavior vary between Adventure levels?
**No.** Calvin is Calvin everywhere. The personality profiles (calvin.ts, nina.ts, rex.ts) are static const objects. There is no per-level difficulty parameter, no skill multiplier, no adventure-specific tuning.

A level specifies `bots: ['beginner', 'beginner']` or `bots: ['intermediate', 'advanced']` — this selects WHICH bot to spawn, but does not modify the bot's behavior within that personality.

### Does behavior change based on game state?
**Yes, via `modifyWeightsForGameState()`** — but only for Rex and Nina:
- Rex shifts to denial mode or conservative mode based on scoreboard (see Q3 above)
- Nina shifts to aggressive mode when trailing (see Q3 above)
- Calvin: no dynamic modification whatsoever

### Other state-based behavior
- `chainPotential` scoring checks remaining hand and board state (evaluator.ts:124-131)
- `opponentDenial` uses CardTracker's `estimateDeckComposition()` to estimate what opponents could pair/sum (evaluator.ts:133-146)
- `jackpotValue` checks `tracker.gamePhase` (late/endgame) and `lastCapturer` status (evaluator.ts:148-156)
- `boardControl` checks how many "partner" cards remain on board for each captured card (evaluator.ts:158-169)

### Is there a skill level or difficulty multiplier parameter?
**No.** The `Difficulty` type (`'beginner' | 'intermediate' | 'advanced'`) selects a personality profile. There is no numeric difficulty slider or multiplier.

## 7. ANYTHING ELSE THAT DIFFERENTIATES THE BOTS

### Thinking delay (visible to player)
Each bot has a different thinking speed, creating a perceptible personality difference:
- Calvin: 1500-3000ms (slowest — "hesitant")
- Nina: 800-1500ms (medium)
- Rex: 400-900ms (fastest — "decisive")

Defined in each personality file's `thinkingDelay: { min, max }`. Used via `getBotThinkingDelay()` in botDecision.ts:69-75, which calls `prng.nextInt(min, max)`.

### Player colors (visible to player)
Bot zones on the gameplay screen glow in personality-specific colors when it's their turn:
- Calvin: blue #3B82F6
- Nina: pale ice #DBEAFE
- Rex: red #DC2626

Defined in `PLAYER_COLORS` in GameView.tsx.

### Bot turn order randomization
Bot seating (which bot plays 2nd vs 3rd) is randomized at game start via seeded PRNG in `createInitialState()` at gameState.ts:69. A 50/50 coin flip swaps `bot1Personality` and `bot2Personality`. This means the same two bots can play in different order each game, affecting strategy (denial bots playing right after vs. right before the player).

### Face-down hand display
Bot zones show face-down card-back hands using the canonical CardComponent at small size. The number of visible card backs matches the bot's actual hand count, so the player can see how many cards each bot holds.

### Nothing else
No bot-specific animations, no move-confidence indicators, no "personality quirks" beyond what's listed above. The bots are functionally identical except for their weight vectors, feature flags (multi-slot, chain eval, placement preference), mistake rates, and thinking speeds.
